# 1. 目的

ユーザーがエディター管理ダイアログにおいて「削除」を実行した際、システム状態とUI状態の乖離（適用ボタンを押すまでは実際には削除されていないにもかかわらず、UI上から直ちに消えること）による誤解を防ぐ。

## 解決すべき課題

- 行が即座に消えることで、「削除完了」と誤認される。
- 適用せずに閉じてしまい、意図した変更が失われる。

## 提案された解決策

以下の3つのアプローチを組み合わせて実装する。
1. **案2:** 削除ボタンを押した際に行を隠すのではなく、「削除マーク」をつける。
2. **代替案A:** 入力欄をグレーアウト（無効化）し、「削除」ボタンを「元に戻す」に変更する。
3. **代替案B:** ダイアログを閉じる際に、未保存の変更（削除や編集など）があれば警告を出す。

# 2. 変更内容

## 2.1 `EditorRow` クラスの改修
- `_delete` メソッドを `_toggle_delete` に変更し、削除フラグのON/OFFでUI状態（`state="normal"` / `state="disabled"`）を切り替えるようにする。
- 削除状態の場合は、警告ラベルに「⚠ 「適用」をクリックすると削除されます」と表示する。

## 2.2 `show_editor_manager` 関数の改修
- ダイアログ初期化時に、現在の設定（`initial_editors = config_manager.get_custom_editors()`）を保持する。
- ウィンドウを閉じる操作（Xボタン、または「閉じる」ボタン）をフックする `on_closing` を実装。
- 閉じる際に現在のUI状態（`current_editors`）を `initial_editors` と比較し、差異があれば `messagebox.askyesno` で警告を表示する。
- 適用（`on_apply`）が行われた際は、`initial_editors` を最新状態に更新する。
